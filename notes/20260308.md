# Sglang 的异步调度是如何实现的

维持两个 batch 。推理 bA 存 CPU 结果；处理 bB 的结果，更新请求状态及流式输出；组织新的 bB 然后推理。

- Q: 异步推理怎么判断请求是否终止，继而组织新的输入批？
- A: 比如某个请求再 bB 推理的时候会命中 eos ，那么在异步组织 bA 的时候认为所有请求都没有结束。直到下一轮处理 bB 发现 eos 再标记结束。

- Q: 在 decode 阶段，下一次输入依赖前一次的输出。在前一次输出没有同步结果到 CPU 的情况下，怎么组织下一次输入？
- A: CPU 测只需要用一个占位符就好了，比如 -1 。在 CPU 侧组织输入并不需要读取这个 token 具体是多少。新组成的 batch 在推理前把这个占位符替换为那个未同步的 token （一个张量）就可以。


# 除了实现 PD 分离和全局 KVCache 这两个大的功能，还有什么别的优化

1. 实现了 D 节点的负载均衡，社区目前只有轮询的版本。
2. 实现了 global bs 和 local bs 的切换。
  - 社区采用 global bs ，每个 tp 用不同的 mask 来避免无用计算，但是 mask 只被用在了 MLP 部分，其他的计算比如 RMSNorm / 残差 都没有使用 mask ，导致性能下降。
  - 我们采用了 local bs 的最大值，在负载均衡的条件下基本不怎么会产生无效计算。
3. 模型结构做了一些修改，比如 DeepSeek-V3 使用 w8a8 的时候， MLP 部分有先量化再反量化的操作，这里社区实现的很差，产生一系列小 kernel ，我们优化了这块。
4. 跑了一些 GEMM 在 H20 上的配置。


# 对比 MOE 架构的 TP / EP 通信量

开了 DP 每张卡都只有自己的数据。

- K: topk 的值
- H: hidden_state 的大小
- P: TP / EP 的并发大小

## EP

1. 选取 topk 个专家，也就是说每个令牌有 topk 次矩阵运算。
2. dispatch 分发，此处每张卡每个令牌产生 ``2KH`` 数据量的 all-to-all 通信。
3. MLP 矩阵计算。
4. combine 聚合，此处每张卡每个令牌产生 ``2KH`` 数据量的 all-to-all 通信。
5. topk 专家加权求和。

## TP

1. 所有 rank 收集全局输入，此处每张卡每个令牌产生 ``2PH`` 数据量的 all-gather 通信。
2. 选取 topk 个专家。
3. MLP 矩阵计算。
4. topk 专家加权求和。
5. 所有 rank 合并结果，此处产生 ``2PH`` 数据量的 all-reduce 通信。
