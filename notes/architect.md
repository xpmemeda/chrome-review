# 合作

## 怎么设计接口？

接口尽可能只用稳定的对象，比如 ``std::string``，不要用自定义的类型。稳定接口依赖稳定抽象，不要让可变实现细节穿透边界。

## 组件的设计是越简单越好吗？接口感知的内容越多，后续变化牵扯的合作方也越多。

# 设计

## 复杂性是系统最大的敌人

- 性能问题能优化
- bug 能修
- 机器能加
- 复杂性只能不断累积，很难消除

即使多写点代码也不要引入额外的设定，不要增加复杂性。

如果某个特性，业务不是长期要用，代码不建议合并到主分支。

## 过早的优化是万恶之源（Premature optimization is the root of all evil -- Donald Knuth）

Make it work -> Make it right -> Make it fast

- 优化的目标大概率是错的。
- 优化会提升复杂度，直接降低系统可理解性。如果优化没有取得实际效果，那么因此增加的开发和维护成本是白白浪费的。
- 过早优化会冻结产品形态，使后续需求演进成本显著上升。

## 抽象的价值在于「隐藏变化」，不是「看起来优雅」

这是架构评审里非常核心的一条隐性共识：

- 好抽象：
  - 屏蔽变化
  - 稳定边界

- 坏抽象：
  - 只封装当前实现
  - 变化一来全漏

## 追究细节

一个细节就能引出很多问题，这是完善架构的契机，也是 Demo 变成工业级代码的路径。

Demo 代码往往关注：

- 功能是否能 work
- happy path 是否成立

工业级代码必须回答：

- 异常路径发生时会怎样？
- 资源耗尽时会怎样？
- 并发 / 重入 / 乱序时会怎样？
- 失败是显式的，还是静默的？

可用性（usability & reliability）来自细节，而不是主流程。

## 没有测量，就没有优化

- 直觉几乎总是错的
- profile 往往推翻假设
- 真正的瓶颈通常集中在极少数代码路径

凭感觉做性能架构，在工程界基本被视为不专业。

## 设计中最重要的是「可删除性」

好设计：

- 模块能删
- 优化能关
- 功能能降级

坏设计：

- 一删就系统崩
- 核心路径依赖高级特性
